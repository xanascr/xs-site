Control Flow

XS
SE LIGA SO (age >= 18) {
  SOLTA O GRITO("Adult")
} SENAO SE LIGA SO (age >= 12) {
  SOLTA O GRITO("Teen")
} SENAO {
  SOLTA O GRITO("Child")
}

REPETE NA MORAL (CRIA i = 0; i < 5; i += 1) {
  SOLTA O GRITO(i)
}

REPETE AI (condition) { ... }

VOA()       // break
CONTINUA()  // continue

SE LIGA SO / SENAO SE LIGA SO / SENAO for conditionals.

REPETE NA MORAL = for. REPETE AI = while.

VOA() = break. CONTINUA() = continue.

Classes and OOP

CLASSE Animal {
  CONSTRUTOR(name) {
    CRIA ISTO.name = name

CLASSE Dog HERDA Animal {
  METODO speak() {

  }
}

CRIA rex = NOVA Dog("Rex")
rex.speak()

CLASSE, HERDA, CONSTRUTOR, METODO, ISTO (this), NOVA (new).

Simple inheritance with HERDA. Methods declared with METODO.

Switch / Pattern Matching

XS
ESCOLHE (day) {
  CASO 1:
    SOLTA O GRITO("Sunday")
  CASO 2:
  CASO 3:
    SOLTA O GRITO("Week start")
  PADRAO:
    SOLTA O GRITO("Other")
}

CRIA res = COMBINA (value) {
  CASO 0 => "zero"
  CASO [a, b] => "array of 2"
  CASO { name } => name
  CASO _ => "other"
}

ESCOLHE/CASO/PADRAO — traditional switch with fallthrough.

COMBINA — Rust-style pattern matching. Destructures arrays, objects. _ is wildcard.

Why XanaScript?

A language designed from the ground up with a full compiler toolchain, not a transpiler or wrapper.

Portuguese Syntax

Multi-word keywords (SE LIGA SO, CHAMA ESSE CARA, REPETE NA MORAL) are lexed as atomic tokens via lookahead DFA. No translation layer — the grammar itself is Portuguese.

Optimizing JS Codegen

Multi-pass optimizer: constant folding, loop unrolling (up to 8 iterations), integer type inference for automatic Int32Array/Float64Array usage, and function inlining.

Native WebAssembly

Direct .wasm binary emission — no Emscripten, no wabt.js, no LLVM. Implements Section Type/Function/Code/Export/Data manually with a bundled linker that resolves function indices and label offsets.

Built-in ORM

TABELA is a first-class syntax construct parsed into AST nodes. The codegen emits complete CRUD functions (inserir, buscar, atualizar, remover) with filter support ($gt, $lt, $eq) and JSON persistence.

Compile-Time Macros

MACRO definitions are stored in the symbol table and expanded during AST traversal before codegen. Supports recursive expansion with configurable depth limits. Zero runtime footprint.

Native Test Runner

TESTE and AFIRMA are parsed as AST statements and executed by the interpreter. Auto-discovers test files, outputs TAP-compatible results, and exits with CI-ready codes.

LSP Server

Language Server Protocol over stdio. Stateless architecture: each request re-parses the document from scratch. Supports completions, hover docs, go-to-definition, real-time diagnostics, and formatting.

Package Manager

xs install resolves packages from GitHub repositories with semver compatibility. xs publish creates a registry entry. npm fallback for JS interop.

Rust-Style Errors

Each error includes a unique code, exact source location with arrow annotation, and structured hint/help messages. The parser implements panic-mode error recovery to report multiple errors per run.

VS Code Extension

Uses vscode-xs with TextMate grammars for syntax highlighting, 18 code snippets, and LSP client integration. Commands for run, build, test, and format.

Complete CLI

20+ subcommands: run, build (JS/Wasm), test, repl, format, docs, lsp, init, install, publish, watch, benchmark. Single binary via bun --compile, zero runtime deps.

Task Runner

TAREFA declarations in tarefas.xs define named tasks parsed into AST. Executed via xs <taskname> with dependency resolution.

Native WebAssembly

Direct .wasm emission — no Emscripten, no wabt.js, no LLVM toolchain.

XS
CHAMA ESSE CARA sum(a, b) {
  VOLTA a + b
}
CHAMA ESSE CARA main() {
  VOLTA sum(10, 20)
}
$ xs build --wasm math.xs

wasm_binary.js implements the Wasm binary format specification from scratch:

  • Magic number (00 61 73 6d) + version (01 00 00 00)
  • Section encoding: Type Section (function signatures), Function Section (type index mapping), Code Section (function bodies with local declarations), Export Section (named exports), Data Section (initial memory contents)
  • All integers encoded via LEB128 (unsigned and signed variants)
  • Two-value type system: i32 and f64 — all XS arithmetic operations map to the appropriate opcode

The internal linker resolves function indices between sections during a second pass. After codegen, it patches label offsets for branch instructions (br, br_if, block, loop). The linker also handles export name string table encoding.

Memory allocation uses a bundled bump allocator — CRIA VETOR with compile-time known sizes emits memory.grow + pointer arithmetic. Strings are stored as (length, data) pairs at the allocated offset.

Built-in ORM

Tables as a first-class AST node — parsed, type-checked, and codegen'd in the compiler pipeline.

XS
TABELA Product {
  name: TEXTO,
  price: NUMERO,
  stock: NUMERO
}

CRIA repo = Product
repo.create({ name: "Keyboard", price: 250, stock: 10 })
repo.create({ name: "Mouse", price: 120, stock: 25 })

SOLTA O GRITO(repo.list())          // all
SOLTA O GRITO(repo.find(1))        // by ID
CRIA expensive = repo.findWhere({ price: 250 })
repo.update(1, { name: "RGB Keyboard" })
repo.delete(2)

TABELA is parsed as a TableDef AST node containing a name and typed field list. The parser validates that field types are one of the five supported: TEXTO, NUMERO, BOOLEANO, DATA, QUALQUER. Invalid types produce a compile-time error with source annotation.

On the codegen side, TableDef emits a constructor function that returns an object with the CRUD methods. The runtime stores data as JSON files on disk (Node/Bun) or in-memory (browser). Each record gets an auto-incrementing _id field plus _criado and _atualizado timestamps.

The findWhere method supports filter operators passed as keys prefixed with $: $eq, $gt, $lt, $gte, $lte, $ne, and $in (array match). Compound filters use implicit AND semantics.

Compile-Time Macros

AST-level substitution — expanded during the optimization phase, erased before codegen.

XS
MACRO square(x) { x * x }
MACRO cube(x) { x * square(x) }
MACRO max(a, b) {
  SE LIGA SO (a > b) { a } SENAO { b }
}
CRIA r1 = square(5)      // -> 5*5 = 25
CRIA r2 = cube(3)          // -> 27
CRIA r3 = max(10, 20)     // -> 20
CRIA r4 = max(square(4), cube(2))

Macros are parsed into MacroDef AST nodes with a parameter list and body. During the optimization phase, the macro expander walks the AST and replaces each CallExpr matching a macro name with the macro's body AST, substituting parameter identifiers with the argument expressions via a shallow copy.

Expansion is recursive: if a macro body contains calls to other macros, those are expanded in a post-substitution pass. The expander tracks a depth counter (default limit: 64) to prevent infinite recursion from circular macro references. After full expansion, all MacroDef nodes are removed from the AST.

Because expansion operates on the AST rather than text (unlike C preprocessor #define), there is no risk of operator-precedence bugs or unintended token pasting. The resulting AST is then fed through the optimizer and codegen as if the macro bodies were written inline.

Test Runner

First-class TESTE and AFIRMA AST nodes — parsed, interpreted, and reported by the compiler itself.

XS
TESTE "sum of two numbers" {
  CRIA r = 2 + 3
  AFIRMA(r == 5)
  ASSUNTO(r, 5)
}
TESTE "multiplication" {
  CRIA r = 3 * 4
  ASSUNTO(r, 12)
}
$ xs test .

TESTE blocks are parsed into TestStmt AST nodes containing an expression (the description) and a statement block. AFIRMA and ASSUNTO are parsed as AssertStmt nodes — the former takes a single boolean expression, the latter takes an actual/expected pair with structural comparison.

The test runner discovers files by glob pattern (**/*.test.xs) and executes them through the AST interpreter. Each TestStmt is wrapped in a try-catch: assertion failures throw structured AssertionError objects with file/line/diff info. The runner aggregates results and outputs TAP-compatible format with colored pass/fail indicators, execution time per test, and a summary line.

Exit code 1 on any failure — CI/CD ready. The xs test --watch flag re-runs on file changes using the built-in file watcher.

Output
  PASS sum of two numbers
  PASS multiplication
  FAIL division by zero

  2 passed  1 failed  0.01s

LSP — Language Server Protocol

Architecture

Multi-stage pipeline from Portuguese source to optimized bytecode, JavaScript, or WebAssembly.

Source (.xs)
→
Lexer
lexer.js
→
Parser
parser.js
→
Optimizer
opt.js

Lexer — DFA-Based Tokenization

The lexer operates as a deterministic finite automaton scanning left-to-right with one-token lookahead. Multi-word keywords (e.g., SE LIGA SO) are matched by consuming consecutive WORD tokens from the stream and checking a keyword dictionary. Unicode support through full String.codePointAt iteration.

Parser — Recursive Descent with Precedence Climbing

Hand-written recursive-descent parser producing a typed AST with 40+ node types. Expressions use precedence climbing (for +, *, comparisons, etc.) with left-associativity. Implements panic-mode error recovery — on parse error, skips to the next statement boundary and continues reporting.

Optimizer — Multi-Pass AST Rewriting

Runs a series of tree walks: constant folding (pre-computes 2 + 3 * 5 at compile time), loop unrolling (up to 8 iterations for fixed-range REPETE NA MORAL loops), integer hinting (annotates variables as integer-only when no reassignment mixes types), function inlining (for single-call-site functions under 15 nodes).

Codegen — Three Backend Targets

codegen.js: direct JavaScript emission with keyword mapping (CRIA → let, etc.). codegen_opt.js: enhanced JS using TypedArrays (auto Int32Array/Float64Array) and COMBINA pattern-match emission. codegen_wasm.js: emits Wasm binary byte-by-byte with internal linker, supporting i32/f64 locals, branching, memory operations, and export resolution.

Runtime (interpreter): AST-walking interpreter for REPL, test runner, and quick execution. Shares the AST tree with all codegen backends.
