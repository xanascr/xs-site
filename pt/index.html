Controle de Fluxo

XS
SE LIGA SO (idade >= 18) {
  SOLTA O GRITO("Maior")
} SENAO SE LIGA SO (idade >= 12) {
  SOLTA O GRITO("Adolescente")
} SENAO {
  SOLTA O GRITO("Crianca")
}

REPETE NA MORAL (CRIA i = 0; i < 5; i += 1) {
  SOLTA O GRITO(i)
}

REPETE AI (condicao) { ... }

VOA()       // break
CONTINUA()  // continue

SE LIGA SO / SENAO SE LIGA SO / SENAO para condicionais.

REPETE NA MORAL = for. REPETE AI = while.

VOA() = break. CONTINUA() = continue.

Classes e OOP

XS
CLASSE Animal {
  CONSTRUTOR(nome) {
    CRIA ISTO.nome = nome
  }
  METODO falar() {
    SOLTA O GRITO(ISTO.nome + " faz barulho")
  }
}

CLASSE Cachorro HERDA Animal {
  METODO falar() {
    SOLTA O GRITO(ISTO.nome + " faz: Au au!")
  }
}

CRIA rex = NOVA Cachorro("Rex")
rex.falar()

CLASSE, HERDA, CONSTRUTOR, METODO, ISTO (this), NOVA (new).

Heranca simples com HERDA. Metodos sao declarados com METODO.

Switch / Pattern Matching

XS
ESCOLHE (dia) {
  CASO 1:
    SOLTA O GRITO("Domingo")
  CASO 2:
  CASO 3:
    SOLTA O GRITO("Inicio da semana")
  PADRAO:
    SOLTA O GRITO("Outro")

CRIA res = COMBINA (valor) {
  CASO 0 => "zero"
  CASO [a, b] => "array de 2"
  CASO { nome } => nome
  CASO _ => "outro"
}

ESCOLHE/CASO/PADRAO — switch tradicional com fallthrough.

COMBINA — pattern matching estilo Rust. Desestrutura arrays, objetos. _ e wildcard.

Por que XanaScript?

Uma linguagem construida do zero com um pipeline de compilador completo, nao um transpilador ou wrapper.

Sintaxe em Portugues

Keywords multi-palavra (SE LIGA SO, CHAMA ESSE CARA, REPETE NA MORAL) sao lexadas como tokens atomicos via DFA com lookahead. Sem camada de traducao — a gramatica em si e portugues.

JS Ultra-Otimizado

Otimizador multi-passo: constant folding, loop unrolling (ate 8 iteracoes), inferencia de tipo inteiro para uso automatico de Int32Array/Float64Array e inlining de funcoes.

WebAssembly Nativo

Emissao binaria direta de .wasm — sem Emscripten, sem wabt.js, sem LLVM. Implementa as secoes Type/Function/Code/Export/Data manualmente com um linker interno que resolve indices de funcao e offsets de label.

ORM Embutido

TABELA e um construto sintatico de primeira classe parseado em nos AST. O codegen emite funcoes CRUD completas (criar, buscar, atualizar, deletar) com suporte a filtros ($gt, $lt, $eq) e persistencia JSON.

Macros Compile-Time

Definicoes MACRO sao armazenadas na tabela de simbolos e expandidas durante a travessia da AST antes do codegen. Suporta expansao recursiva com limite de profundidade configuravel. Zero custo em runtime.

Test Runner Nativo

TESTE e AFIRMA sao parseados como instrucoes AST e executados pelo interpretador. Descobre arquivos de teste automaticamente, gera resultados compativeis com TAP e sai com codigos prontos para CI.

LSP Completo

Language Server Protocol via stdio. Arquitetura stateless: cada requisicao re-parseia o documento do zero. Suporta completions, hover docs, go-to-definition, diagnosticos em tempo real e formatacao.

Gerenciador de Pacotes

xs install resolve pacotes de repositorios GitHub com compatibilidade semver. xs publish cria uma entrada no registro. Fallback npm para interop JS.

Erros estilo Rust

Cada erro inclui um codigo unico, localizacao exata na fonte com anotacao de seta e mensagens estruturadas de hint/help. O parser implementa panic-mode error recovery para reportar multiplos erros por execucao.

VS Code Extension

Usa vscode-xs com gramaticas TextMate para syntax highlighting, 18 snippets de codigo e integracao com o cliente LSP. Comandos para run, build, test e format.

CLI Completa

20+ subcomandos: run, build (JS/Wasm), test, repl, format, docs, lsp, init, install, publish, watch, benchmark. Binario unico via bun --compile, zero dependencias em runtime.

Task Runner

Declaracoes TAREFA em tarefas.xs definem tarefas nomeadas parseadas em AST. Executadas via xs <taskname> com resolucao de dependencias.

WebAssembly Nativo

Emissao direta de .wasm — sem Emscripten, sem wabt.js, sem LLVM.

XS
CHAMA ESSE CARA soma(a, b) {
  VOLTA a + b
}
CHAMA ESSE CARA main() {
  VOLTA soma(10, 20)
}
$ xs build --wasm math.xs

wasm_binary.js implementa a especificacao do formato binario Wasm do zero:

  • Magic number (00 61 73 6d) + versao (01 00 00 00)
  • Codificacao de secoes: Type Section (assinaturas de funcao), Function Section (mapeamento de indices de tipo), Code Section (corpos de funcao com declaracoes locais), Export Section (exportacoes nomeadas), Data Section (conteudo inicial da memoria)
  • Todos os inteiros codificados via LEB128 (variantes signed e unsigned)
  • Sistema de tipos com dois valores: i32 e f64 — todas as operacoes aritmeticas do XS mapeiam para o opcode apropriado

O linker interno resolve indices de funcao entre secoes durante um segundo passe. Apos o codegen, ele corrige offsets de label para instrucoes de desvio (br, br_if, block, loop). O linker tambem gerencia a codificacao da tabela de strings de nomes de exportacao.

A alocacao de memoria usa um bump allocator interno — CRIA VETOR com tamanhos conhecidos em tempo de compilacao emite memory.grow + aritmetica de ponteiros. Strings sao armazenadas como pares (tamanho, dados) no offset alocado.

i32 / f64Tipos de valor
block / loopFluxo de controle
call / returnFuncoes
memory.growAlocacao dinamica

ORM Embutido

Tabelas como nos AST de primeira classe — parseadas, verificadas por tipo e geradas no pipeline do compilador.

XS
TABELA Produto {
  nome: TEXTO,
  preco: NUMERO,
  estoque: NUMERO
}

CRIA repo = Produto
repo.criar({ nome: "Teclado", preco: 250, estoque: 10 })
repo.criar({ nome: "Mouse", preco: 120, estoque: 25 })

SOLTA O GRITO(repo.listar())       // todos
SOLTA O GRITO(repo.buscar(1))     // por ID
CRIA caros = repo.buscarOnde({ preco: 250 })
repo.atualizar(1, { nome: "Teclado RGB" })
repo.deletar(2)

TABELA e parseado como um no AST TableDef contendo um nome e uma lista de campos tipados. O parser valida que os tipos dos campos sao um dos cinco suportados: TEXTO, NUMERO, BOOLEANO, DATA, QUALQUER. Tipos invalidos produzem um erro em tempo de compilacao com anotacao de origem.

No codegen, TableDef emite uma funcao construtora que retorna um objeto com os metodos CRUD. O runtime armazena dados como arquivos JSON em disco (Node/Bun) ou em memoria (navegador). Cada registro ganha um campo _id auto-incrementado mais timestamps _criado e _atualizado.

O metodo buscarOnde suporta operadores de filtro passados como chaves prefixadas com $: $eq, $gt, $lt, $gte, $lte, $ne e $in (correspondencia em array). Filtros compostos usam semantica AND implicita.

TEXTOstring
NUMEROnumber
BOOLEANOboolean
DATAstring ISO
QUALQUERany

Macros Compile-Time

Substituicao em nivel de AST — expandido durante a fase de otimizacao, apagado antes do codegen.

XS
MACRO quadrado(x) { x * x }
MACRO cubo(x) { x * quadrado(x) }
MACRO maior(a, b) {
  SE LIGA SO (a > b) { a } SENAO { b }
}
CRIA r1 = quadrado(5)      // -> 5*5 = 25
CRIA r2 = cubo(3)          // -> 27
CRIA r3 = maior(10, 20)    // -> 20
CRIA r4 = maior(quadrado(4), cubo(2))

Macros sao parseadas em nos AST MacroDef com uma lista de parametros e um corpo. Durante a fase de otimizacao, o expansor de macros percorre a AST e substitui cada CallExpr que corresponde a um nome de macro pelo corpo da macro, substituindo identificadores de parametros pelas expressoes de argumento via copia superficial.

A expansao e recursiva: se um corpo de macro contem chamadas a outras macros, estas sao expandidas em um passe pos-substituicao. O expansor rastreia um contador de profundidade (limite padrao: 64) para evitar recursao infinita de referencias circulares de macros. Apos a expansao completa, todos os nos MacroDef sao removidos da AST.

Como a expansao opera na AST em vez de texto (diferente do preprocessador C #define), nao ha risco de bugs de precedencia de operadores ou colagem acidental de tokens. A AST resultante e entao alimentada pelo otimizador e codegen como se os corpos das macros tivessem sido escritos inline.

Test Runner

Nos AST TESTE e AFIRMA de primeira classe — parseados, interpretados e reportados pelo proprio compilador.

XS
TESTE "soma de dois numeros" {
  CRIA r = 2 + 3
  AFIRMA(r == 5)
  ASSUNTO(r, 5)
}
TESTE "multiplicacao" {
  CRIA r = 3 * 4
  ASSUNTO(r, 12)
}
$ xs test .

Blocos TESTE sao parseados em nos AST TestStmt contendo uma expressao (a descricao) e um bloco de instrucoes. AFIRMA e ASSUNTO sao parseados como nos AssertStmt — o primeiro recebe uma expressao booleana unica, o segundo recebe um par atual/esperado com comparacao estrutural.

O test runner descobre arquivos por glob pattern (**/*.test.xs) e os executa via interpretador AST. Cada TestStmt e envolvido em um try-catch: falhas de assertion lancam objetos AssertionError estruturados com informacao de arquivo/linha/diff. O runner agrega resultados e gera sauda no formato TAP com indicadores coloridos de passou/falhou, tempo de execucao por teste e uma linha de resumo.

Exit code 1 em qualquer falha — pronto para CI/CD. A flag xs test --watch re-executa ao detectar alteracoes nos arquivos usando o file watcher interno.

Output
  PASS soma de dois numeros
  PASS multiplicacao
  FAIL divisao por zero

  2 passaram  1 falhou  0.01s

LSP — Language Server Protocol

Servidor LSP stateless via stdin/stdout JSON-RPC — um parse por requisicao, sem estado persistente.

Inicie o servidor com xs lsp. Implementa textDocument/completion (itens de completacao de keywords/builtins/identificadores com kind e detail), textDocument/hover (docs Markdown para funcoes built-in com assinatura), textDocument/definition (resolve declaracoes de variaveis/funcoes para sua localizacao na fonte) e textDocument/diagnostic (saida completa de erros do compilador mapeada para Range, incluindo codigo de erro e severidade).

Bash
xs lsp

Neovim

Lua

  pattern = "xs",

    vim.lsp.start({

    })
  end,
})
